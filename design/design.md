# spec-divisor-finder-agent 設計メモ

この文書は「なぜこの設計にしたか」の記録。使い方は README、概念入門は 01_airepo の `00_入門_…`／`00_詳説_…`。

## 実証する設計パターン
**仕様（アサーション）オラクル**。出力が一意に決まらない処理を、保存した正解と比べるのではなく、
**出力が満たすべき仕様**を機械的に検証して合否を決める。

## なぜこの方式を選んだか（他方式との比較）★この repo の肝
- 「n の約数を1つ返す」は**答えが一意でない**（12 → 2 でも 3 でも 6 でも正しい）。だから golden（出力一致）も差分テストも使えない。
- だが**正しさの条件は完全に言葉にできる**：返した d は 1<d<n かつ割り切る／None なら本当に素数。これを**オラクルが独立に再計算して**照合する。
- ポイントは「答えを知らなくても、答えの正しさは検証できる」こと（検証は生成より易しい）。これが仕様オラクル。
- 対比：golden/差分＝正解が1つに決まる時。メタモルフィック＝入力間の関係が言える時。本 repo は「**正解は一意でないが、正しさの条件は厳密に書ける**」列を担当。

## 仕様（オラクルが照合する条件）
- 約数を返した: `1 < d < n` かつ `n % d == 0`。
- None を返した: n が素数（オラクル側で試し割りして確認）。

## なぜ信頼できるか（オラクル自己検証）
`--selftest` で、正しい実装は PASS、既知バグ（常に 2／n 自身を返す／平方数を素数と誤判定）はそれぞれ別の仕様違反として FAIL になることを示す。

## 既知の限界
- テスト範囲は有限（n=2..5000）。範囲外の極端な n は見ていない。
- 仕様は「非自明な約数を1つ」。最小である必要や全約数の列挙までは要求していない（そこは別仕様）。
